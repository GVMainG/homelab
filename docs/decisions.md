# Архитектурные решения (ADR)

## Для чего этот документ и как вести?

Этот документ фиксирует **почему** были приняты те или иные технические решения. Когда через полгода возникает вопрос «почему мы используем именно это?», ответ будет здесь — с контекстом, альтернативами и обоснованием выбора.

**Что должно быть в этом документе:**

Для каждого архитектурного решения:

- **Дата** — когда решение было принято
- **Контекст** — какая проблема решалась
- **Варианты** — какие альтернативы рассматривались
- **Решение** — что выбрано и почему
- **Последствия** — что это дало, какие появились ограничения
- **Статус** — `accepted` / `superseded` / `deprecated`

Примеры решений для документирования:

- Почему Vaultwarden, а не официальный Bitwarden
- Почему PostgreSQL 16, а не 17 или MySQL
- Почему sparse checkout, а не отдельные репозитории
- Почему Nginx Proxy Manager, а не Traefik или Caddy
- Почему Proxmox, а не VMware или KVM без оркестрации
- Почему dnsmasq для split-DNS, а не Pi-hole или Unbound

**Правила документирования:**

- Каждое решение — отдельный подраздел с датой в заголовке (например: `## 2025-04-14 — Выбор Vaultwarden`)
- Писать кратко, но с достаточным контекстом, чтобы понять логику
- Указывать конкретные альтернативы, которые рассматривались (не просто «другие варианты хуже», а «Traefik не подошёл потому что…»)
- Обновлять статус решения, если оно пересмотрено
- Не удалять старые решения — помечать как `deprecated` с ссылкой на новое
- Ссылаться на этот документ из `changelog.md` при фиксации изменений

## 2026-04-14 — Git sparse checkout для каждой VM

- **Контекст:** Несколько VM должны управлять своей конфигурацией из одного репозитория, но клонировать весь репозиторий избыточно и небезопасно.
- **Варианты:**
  - Отдельные репозитории на каждую VM — дублирование кода, сложно синхронизировать общие паттерны
  - Полная клонирование репозитория на каждую VM — лишние файлы, риск утечки конфигов других VM
  - Git sparse checkout — одна репа, каждая VM видит только свой подкаталог
- **Решение:** Использовать `git clone --filter=blob:none --sparse` + `git sparse-checkout set <vm-name>`. Скрипт `sync.sh` автоматизирует обновление.
- **Последствия:** Каждая VM изолирована, но общая инфраструктура (конвенции, скрипты, документация) — в одном месте. Минус: нужно помнить про sparse checkout при работе с репо локально.
- **Статус:** accepted

## 2026-04-14 — Упрощение архитектуры: одна VM вместо пары

- **Контекст:** Ранее использовалась пара vm-db-02 + vm-proxy-02 (сервисы за reverse proxy с SSL). Для домашнего использования это оказалось избыточно — NPM, dnsmasq, Homepage добавляли сложность обслуживания.
- **Варианты:**
  - Оставить пару VM с NPM/SSL — больше безопасности, но больше поддержки
  - Одна VM, сервисы напрямую по HTTP — проще, но нет HTTPS
- **Решение:** Одна VM (vm-db-01) с PostgreSQL, Vaultwarden, pgAdmin. Сервисы доступны по LAN напрямую: `http://192.168.1.36:8080` (Vaultwarden), `http://192.168.1.36:5050` (pgAdmin).
- **Последствия:** Нет SSL-терминации, нет wildcard DNS. Упрощение обслуживания — один сервер для бэкапа и обновлений.
- **Статус:** accepted

## 2026-04-14 — PostgreSQL на vm-db-01: проброс порта на LAN

- **Контекст:** На vm-db-01 PostgreSQL был привязан только к `127.0.0.1`. На vm-db-02 порт `5432` проброшен на LAN.
- **Варианты:**
  - `127.0.0.1:5432` — только локальный доступ (как на vm-db-01)
  - `0.0.0.0:5432` — доступ со всей LAN
- **Решение:** Порт `5432:5432` проброшен на LAN. Это позволяет подключаться с vm-proxy-02 и других машин для бэкапов, администрирования через pgAdmin с LAN.
- **Последствия:** PostgreSQL доступен из LAN — необходимо обеспечить защиту паролем и фаерволом на уровне ОС.
- **Статус:** accepted

## 2025-xx-xx — Выбор Vaultwarden вместо официального Bitwarden

- **Контекст:** Нужен менеджер паролей для домашнего использования.
- **Варианты:**
  - Официальный Bitwarden server — тяжёлый, требует MSSQL/SQL Server, много ресурсов
  - Vaultwarden — лёгкий, написан на Rust, поддерживает тот же API, работает с PostgreSQL
- **Решение:** Vaultwarden (`vaultwarden/server:latest`) — потребляет меньше ресурсов, совместим с клиентами Bitwarden.
- **Последствия:** Не все фичи официального сервера доступны, но для домашнего использования достаточно.
- **Статус:** accepted

## 2025-xx-xx — Выбор Nginx Proxy Manager вместо Traefik/Caddy

- **Контекст:** Нужен reverse proxy с SSL-терминацией и простым управлением.
- **Варианты:**
  - Traefik — автоматическое обнаружение сервисов, но сложная конфигурация
  - Caddy — автоматический HTTPS, но меньше гибкости в настройке upstream
  - Nginx Proxy Manager — веб-UI, простая настройка reverse proxy и Let's Encrypt
- **Решение:** Nginx Proxy Manager — веб-интерфейс упрощает добавление новых прокси и SSL-сертификатов.
- **Последствия:** Образ использует тег `latest` (нет стабильных релизов), но это осознанный компромисс.
- **Статус:** accepted

## 2025-xx-xx — Выбор Proxmox VE как гипервизора

- **Контекст:** Нужна платформа для виртуализации homelab.
- **Варианты:**
  - VMware ESXi — проприетарный, ограниченная бесплатная версия
  - KVM без оркестрации — ручное управление VM
  - Proxmox VE — open-source, веб-UI, снапшоты, бэкапы
- **Решение:** Proxmox VE — бесплатный, с веб-интерфейсом, снапшоты VM (`qm snapshot`), встроенные бэкапы.
- **Последствия:** Привязка к экосистеме Proxmox, но для homelab это преимущество.
- **Статус:** accepted

## 2025-xx-xx — dnsmasq для split-DNS

- **Контекст:** Нужен локальный DNS для домена `*.home.loc`, направляющий запросы на proxy VM.
- **Варианты:**
  - Pi-hole — DNS + ad blocking, но тяжелее для простой задачи
  - Unbound — рекурсивный DNS, сложнее для split-DNS
  - dnsmasq — лёгкий, простой конфиг для wildcard DNS
- **Решение:** dnsmasq — минимальный footprint, простая настройка wildcard-записей.
- **Последствия:** Требует остановки `systemd-resolved` на VM, но это разовая операция.
- **Статус:** deprecated (proxy VM удалена 2026-04-14, dnsmasq больше не используется)
