# docs-writer

## Описание

Скилл для автоматического заполнения документации в папке `docs/` на основе:
- коммитов текущей сессии
- истории разговора (что обсуждалось, что менялось)
- текущего состояния файлов в репозитории

## Вызов

Команда: `/docs-u`

## Правила

1. **Не выдумывать** — писать только то, что подтверждено коммитами, файлами или историей чата.
2. **Точные данные** — версии, IP, пути, даты брать из реальных файлов и `git log`.
3. **Сохранять структуру** — каждый документ имеет заголовок «Для чего этот документ и как вести?» — НЕ МЕНЯТЬ его.
4. **Формат** — Markdown, таблицы где уместно, блоки кода с указанием языка.
5. **Хронология** — в changelog записи сверху новые, даты в формате `YYYY-MM-DD`.

## Порядок работы

### Шаг 1 — Сбор контекста

```bash
# Последние коммиты (все, не только текущая сессия)
git log --oneline -20

# Детали коммитов (что именно менялось)
git log --stat -10

# Текущие изменения (ещё не закоммиченные)
git status

# Diff staged файлов
git diff HEAD
```

Прочитать `CLAUDE.md` или `QWEN.md` для понимания конвенций.
Прочитать все файлы в `docs/` для понимания текущего состояния.

### Шаг 2 — Определение что обновлять

На основе коммитов и чата определить, какие документы затронуты:

| Изменение в коде | Документ для обновления |
|---|---|
| Новая VM, новый сервис | `overview.md`, `services.md`, `changelog.md` |
| Изменение конфига сервиса | `services.md` |
| Новый файл/скрипт | `runbooks.md`, `services.md` |
| Смена версии | `services.md`, `changelog.md` |
| Инцидент / фикс | `troubleshooting.md`, `changelog.md` |
| Новое архитектурное решение | `decisions.md` |
| Настройка VM с нуля | `vm-setup.md` |

### Шаг 3 — Заполнение документов

Для каждого затронутого документа:

- **overview.md**: обновить таблицы VM, сервисов, endpoint'ов
- **vm-setup.md**: добавить шаги, если появились новые зависимости
- **services.md**: добавить/обновить секцию сервиса с реальными данными из `docker-compose.yml` и `.env.example`
- **decisions.md**: если в коммитах или чате видно обоснование выбора — записать как ADR
- **changelog.md**: добавить записи по каждому коммиту/изменению в формате:
  ```
  ### YYYY-MM-DD — Краткое описание

  - **Тип:** added|changed|fixed|removed|upgraded
  - **VM:** имя VM
  - **Описание:** что изменилось
  - **Версии:** если менялись версии
  ```
- **runbooks.md**: если в коммитах есть новые скрипты или процедуры — добавить runbook
- **troubleshooting.md**: если в чате обсуждалась проблема и решение — добавить записью

### Шаг 4 — Проверка

- Убедиться, что ни один «Для чего этот документ…» заголовок не изменён
- Проверить форматирование Markdown
- Убедиться, что нет `.env` секретов в документации
- Проверить, что все ссылки на файлы и пути соответствуют реальной структуре

## Пример запроса и выполнения

**Пользователь:** «Обнови документацию по последним изменениям»

**Действия:**
1. `git log --oneline -10 && git status && git diff HEAD`
2. Прочитать `docs/*.md`
3. Прочитать изменённые файлы
4. Определить затронутые документы
5. Внести правки через `edit`
6. Сообщить пользователю что обновлено
